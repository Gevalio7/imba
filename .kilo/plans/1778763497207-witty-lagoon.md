# План: Внедрить проверку прав (показ иконок и серверные запреты) для всех справочников

Цель
- Для всех справочников (menu_*), которые входят в конфиг ролей/разрешений, обеспечить:
  1) Отображение UI-элементов (иконок редактирования/удаления/создания) только при наличии соответствующих прав у текущего пользователя;
  2) Серверную защиту CRUD-эндпойнтов (protect + checkPermission) так, чтобы операции write/delete возвращали 403 при отсутствии прав;
  3) Мгновенную/предсказуемую синхронизацию прав в текущей вкладке после изменения ролей (по возможности — без принудительного перелогина).

Область охвата
- Все категории из `menuConfig` (см. src/pages/apps/roles/edit.vue) включая, но не ограничиваясь: menu_tickets (уже частично сделано), menu_queues, menu_types, menu_type_categories, menu_states, menu_priorities, menu_sla, menu_templates, menu_template_queues, menu_workflows, menu_greetings, menu_signatures, menu_auto_responses, menu_attachments и т.д.

Основные шаги (высокоуровневые)
1) Инвентаризация: найти frontend-страницы/компоненты для каждого справочника (list, add, edit, details) и соответствующие backend routes/controllers.
2) Frontend: для каждой страницы добавить условное отображение действий (Создать/Редактировать/Удалить) используя $can('write'|'delete','menu_xxx') или $can('read','menu_xxx') как нужно; добавить meta (action/subject) на route-страницы, чтобы guard мог проверять доступ.
3) Backend: в файлах routes/*.js (каждая сущность) добавить express middleware protect + checkPermission('menu_xxx_write'|'menu_xxx_delete') на POST/PUT/DELETE маршруты; убедиться, что контроллеры возвращают 403 при недостатке прав.
4) Roles UI: при успешном сохранении прав роли — если изменяемая роль совпадает с ролью текущего пользователя — получать и применять новые userAbilityRules в sessionStorage и обновлять ability/permissionsMap (avoid forcing logout).
5) Router guards: оставить/усилить текущую асинхронную проверку прав (canNavigate) с ретраями при временных ошибках, чтобы случайные 401/403 не приводили к преждевременной очистке sessionStorage.
6) Тесты и валидация: написать чек-лист ручного тестирования и интеграционные тесты для ключевых end‑to‑end сценариев.
7) Rollout и мониторинг: как выкатывать и контролировать изменение на проде.

Детальная реализация (шаг за шагом)

Шаг A — Инвентаризация (1–2 часа)
- Составить список всех menu_* кодов из menuConfig (src/pages/apps/roles/edit.vue).
- Найти фронтенд страницы для каждой категории: обычно src/pages/apps/<entity>/*.vue (list/add/edit). Использовать glob-поиск по названиям маршрутов/файлов.
- Найти backend routes в backend/routes/*.js и контроллеры в backend/controllers/*Controller.js.

Шаг B — Frontend изменения (4–8 часов)
Для каждой найденной сущности выполнить:
1. В list.vue (страница списка):
   - Заменить отображение действий на шаблоне (IconBtn, VBtn и т.д.) так, чтобы:
     - Кнопка Создать: v-if="$can('write','menu_<code>_create'?)" или v-if="$can('write','menu_<entity>')" в зависимости от существующей naming convention. Предпочтение: subject следует брать как тот же код, что используется в меню (menu_<...>_create). Если в навигации create использует subject 'menu_xxx_create' — проверять по нему.
     - Кнопка Редактировать: v-if="$can('write','menu_<entity>')" (например menu_tickets_list)
     - Кнопка Удалить: v-if="$can('delete','menu_<entity>')"
   - Пример (tickets done): IconBtn v-if="$can('delete','menu_tickets_list')" ...
2. В add.vue (форма создания): добавить definePage meta: action: 'write', subject: 'menu_<entity>_create' (если route используется) и на уровне компонента проверять доступ при mount: if (! $can('write','...')) redirect/not-authorized.
3. В edit.vue (форма редактирования): оставить meta action: 'read' для доступа к просмотру, но на кнопке Save поставить v-if="$can('write','menu_<entity>')".
4. В read/view.vue (если нужен отдельный view): meta action:'read', subject:'menu_<entity>' и link/view icon visible when read true.

Примечание naming: в проекте используются комбинации menu_tickets_list, menu_tickets_create, menu_tickets_schedules и суффиксы _read/_write/_delete. Для проверки UI использовать именно те subject-ы, которые присутствуют в menuConfig.

Шаг C — Router/meta (30–60 минут)

Шаг D — Backend: защита маршрутов (1–3 часа)
- Для каждого backend route файла (backend/routes/*.js) проверить POST/PUT/DELETE и добавить protect + checkPermission(...):
  - POST (create) -> checkPermission('menu_<entity>_create_write') или checkPermission('menu_<entity>_write') в зависимости от кодировки проекта (следуйте существующей convention; в ticketSchedules я поставил menu_tickets_schedules_write).
  - PUT (update) -> checkPermission('menu_<entity>_write')
  - DELETE -> checkPermission('menu_<entity>_delete')
- Обеспечить, чтобы middleware permissions.checkPermission корректно возвращал 403 при отсутствии прав и не возвращал 401 (401 только для неавторизованного токена в protect).
- Проверить все endpoints, которые создают/удаляют/изменяют записи (включая вспомогательные run/clone endpoints).
