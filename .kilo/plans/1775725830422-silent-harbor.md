# План рефакторинга создания/редактирования тикетов

## Анализ проблем

### Frontend (add.vue, edit.vue)
- **~20 HTTP запросов** при загрузке страницы (14 для справочников + 6 для тикета)
- Каждая fetch-функция делает отдельный запрос
- Нет кэширования данных между страницами

### Backend (ticketsController.js)
- `updateTicket`: загружаются ВСЕ справочники для истории (строки 557-582), даже если меняется только title
- Запись истории делается последовательно для каждого поля (строки 704-722)
- `createTicket`: много последовательных запросов для автоназначений и SLA

## План рефакторинга

### Этап 1: Бэкап (выполнить перед любыми изменениями)
```bash
# Создание архива проекта
tar -czvf ticket-perf-backup-$(date +%Y%m%d_%H%M%S).tar.gz \
  src/pages/apps/tickets/ \
  backend/controllers/ticketsController.js \
  backend/routes/tickets.js
```

### Этап 2: Backend - Batch endpoint для справочников
1. Создать `backend/routes/referenceData.js` с единым endpoint `/api/reference-data`
2. Возвращает все справочники одним запросом:
   - priorities, queues, states, types, categories, agents, agentGroups, customers, services, sla, systemConfiguration
3. Добавить кэширование на уровне endpoint (5 минут)

### Этап 3: Backend - Оптимизация updateTicket
1. **Загрузка только нужных справочников**: вместо загрузки всех 9 справочников, загружать только те, чьи поля реально изменились
2. **Batch insert истории**: использовать bulk insert вместо цикла
3. Опционально: кэшировать справочники в памяти приложения

### Этап 4: Frontend - Композиable для справочников
1. Создать `src/composables/useReferenceData.ts`:
   - Загрузка справочников через batch endpoint
   - Кэширование в reactive state
   - Методы: `fetchAll()`, `fetchIfNeeded()`

### Этап 5: Frontend - Применение в add.vue и edit.vue
1. Заменить все отдельные fetch на использование `useReferenceData()`
2. Изменить onMounted для использования единого batch запроса
3. Ожидаемое улучшение: с 20 HTTP запросов до 1-2

## Безопасность рефакторинга

1. **Сначала добавляем новый код**, не удаляя старый
2. **Параллельное существование**: старые endpoints работают, новые добавляются
3. **Поэтапное тестирование**: после каждого этапа проверять работоспособность
4. **Rollback план**: бэкап позволяет откатиться
