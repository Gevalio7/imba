13: ## Рекомендации по реализации (добавлено)
14: 
15: - Dynamic interval: после изменения интервала через UI новый интервал должен примениться не позднее, чем через старый интервал + задержка. Реализация с `clearInterval` внутри `runMailFetcher` это решает.
16: - Retry: используем экспоненциальный backoff: `delay = baseDelaySec * 1000 * Math.pow(2, attempt - 1)`.
17:   Пример при `maxRetries=3`, `baseDelaySec=5`: attempt1=5s, attempt2=10s, attempt3=20s — суммарно 35s. Если все попытки исчерпаны — НЕ пробрасывать ошибку выше; логировать в `mail_fetch_logs.errors` и переходить к следующему аккаунту.
18: - Attachment saving: перед реализацией открыть `backend/models/attachments.js` и проверить имя поля и ожидаемый тип (`Buffer` или `base64`). Если модель требует base64 — конвертировать `att.content.toString('base64')`.
19:   Проверять `att.size` против `mail_fetcher_max_attachment_size_mb * 1024 * 1024`; если превышает — пропускать и логировать предупреждение (не как ошибка).
20: - SystemLog UI: в списке обрезать `errors` до 100 символов (`errors_preview`) и показывать tooltip; по клику на строку открывать модалку с полным текстом `errors`. По умолчанию сортировка — `started_at DESC` (свежие сверху). Обеспечить пагинацию и сортировку на бекенде.
21: - Добавить SQL вставку дефолтных конфигов (см. раздел "Системные конфиги").

25: ## Детали изменений
26: 
27: ### 1 · Динамический интервал — `backend/server.js`
28: 
29: Проблема: `getMailFetchIntervalMinutes()` вызывался один раз при старте; после изменения в UI нужен рестарт.
30: 
31: Подход: читать конфиг внутри цикла/схедулера на каждой итерации. При старте вызывать `runMailFetcher()` сразу (без setTimeout), интервал установится после первого выполнения. Это гарантирует, что после изменения интервала новый интервал вступит в силу не позднее старого интервала + задержка.
32: 
33: Пример (псевдо):
34: 
35: ```js
36: let mailFetcherInterval = null;
37: async function runMailFetcher() {
38:   const minutes = await getMailFetchIntervalMinutes(); // читаем при каждой итерации
39:   const ms = (minutes || 5) * 60 * 1000;
40:   if (mailFetcherInterval) clearInterval(mailFetcherInterval);
41:   mailFetcherInterval = setInterval(runMailFetcher, ms);
42:   try {
43:     console.log('[MAIL-FETCHER] run (interval:' + minutes + 'min)', new Date().toISOString());
44:     await MailFetcherService.fetchAllAccounts();
45:   } catch (err) {
46:     console.error('[MAIL-FETCHER] error', err);
47:   }
48: }
49: // start immediately, интервал будет переустановлен внутри
50: runMailFetcher();
51: ```

55: ### 2 · Retry logic — `backend/services/MailFetcherService.js`
56: 
57: Проблема: временные сетевые/IMAP ошибки прерывают обработку аккаунта.
58: 
59: Подход: оборачивать только IMAP-операции (`connect`, `fetch`/lock) в общую функцию `retry(fn, maxRetries, baseDelaySeconds)`. Значения берём из `system_configurations`.
60: 
61: Псевдо-реализация `retry`:
62: 
63: ```js
64: async function retry(fn, maxRetries = 3, baseDelaySec = 5) {
65:   let lastErr;
66:   for (let attempt = 1; attempt <= maxRetries; attempt++) {
67:     try { return await fn(); }
68:     catch (e) {
69:       lastErr = e;
70:       console.warn('[MAIL-FETCHER] retry', attempt, '/', maxRetries, e.message);
71:       const delay = baseDelaySec * 1000 * Math.pow(2, attempt - 1); // экспоненциальный backoff
72:       await new Promise(r => setTimeout(r, delay));
73:     }
74:   }
75:   throw lastErr;
76: }
77: ```
78: 
79: Пример задержек при `maxRetries=3`, `baseDelaySec=5`: 5s, 10s, 20s. После исчерпания попыток — логировать в `mail_fetch_logs.errors` и `continue` к следующему аккаунту (не пробрасывать ошибку выше).

83: ### 3 · Сохранение вложений — `backend/services/MailFetcherService.js` (+ проверка модели)
84: 
85: `mailparser` возвращает `parsed.attachments` — массив `{ filename, contentType, size, content: Buffer }`.
86: 
87: Подход:
88: - Перед сохранением проверить поле `data` в модели `backend/models/attachments.js`: ожидает ли модель `Buffer` или base64. Если модель ожидает base64 — использовать `att.content.toString('base64')`, иначе передавать `Buffer`.
89: - Добавить проверку размера вложения: если `att.size` больше `mail_fetcher_max_attachment_size_mb` — пропустить и записать предупреждение в лог (по умолчанию 10 MB). Не считать это ошибкой для прерывания обработки.
90: - После создания тикета сохранять вложения в `attachments`, затем записывать связь в `ticket_attachments`.
91: 
92: Псевдо:
93: ```js
94: if (parsed.attachments && parsed.attachments.length) {
95:   for (const att of parsed.attachments) {
96:     if (att.size > maxAttachmentBytes) {
97:       // записать warning в mail_fetch_logs.errors или отдельное поле warnings
98:       continue;
99:     }
100:     const dataField = attachmentsModelExpectsBase64 ? att.content.toString('base64') : att.content;
101:     const saved = await Attachments.create({
102:       fileName: att.filename,
103:       contentType: att.contentType,
104:       size: att.size,
105:       data: dataField,
106:       ticketId: created.id,
107:       uploadedBy: ownerId || null,
108:     });
109:     await TicketAttachments.create({ ticketId: created.id, attachmentId: saved.id });
110:   }
111: }
112: ```
