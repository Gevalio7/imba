# План интеграции системы разрешений

## Оставшиеся задачи

### 1. Добавление флага super_user
Добавить `super_user` в список разрешений, чтобы супер-пользователь мог иметь полный доступ.

### 2. Интеграция проверки разрешений в контроллеры тикетов
Модифицировать `backend/controllers/ticketsController.js`:
- `getTickets` - фильтровать тикеты по уровню доступа (свои/отдела/компании/все)
- `createTicket` - проверять разрешение `create_ticket`
- `updateTicket` - проверять разрешения `change_status`, `reply_to_tickets`
- `changeTicketStatus` - проверять разрешение `change_status`

### 3. Привязка роли к сервису (новая функциональность)
Согласно вашему требованию: "Агент может работать только с сервисами указанными в очереди, роль нужно привязывать к сервису"

Текущая архитектура:
- Agents → role_id → Roles
- Queues → serviceId → Services
- Agents → agents_groups → AgentsGroups

Предлагаемое решение:
- Создать таблицу `role_service_permissions` для привязки ролей к сервисам
- Роль может иметь разрешения для определённых сервисов

## Вопросы для уточнения

1. **Флаг super_user**: Как должен работать super_user?
   - [ ] Игнорирует все проверки разрешений
   - [ ] Имеет доступ ко всему, но с логированием
   - [ ] Другое

2. **Привязка роли к сервису**: Как это должно работать?
   - [ ] Роль + Сервис = набор разрешений (агент имеет разные разрешения для разных сервисов)
   - [ ] Сервис имеет свою роль по умолчанию
   - [ ] Агент может работать только с сервисами очередей, на которые он назначен

3. **Уровни доступа к тикетам**: Кто может видеть какие тикеты?
   - Агент с `see_own_tickets` - только свои
   - Агент с `see_department_tickets` - тикеты отдела (нужно определить как определяется отдел)
   - Агент с `see_company_tickets` - все тикеты компании
   - Агент с `see_all_tickets` - все тикеты системы
